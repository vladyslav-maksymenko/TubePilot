# Інтеграція Google Drive (TubePilot)

## 📌 Огляд Архітектури (Overview)
Цей модуль відповідає за автоматичний моніторинг (polling) специфічної папки Google Drive. Він знаходить нові відеофайли, завантажує їх локально і веде реєстр вже завантажених файлів, щоб уникнути їх повторного виконання.

Система складається з трьох ключових компонентів:
1. **`GoogleDriveWatcher`** (Сервіс підключення): Встановлює аутентифікацію через Service Account, формує пошуковий запит до Google Drive API (шукає виключно `mimeType contains 'video/'`) та стягує файли чанками.
2. **`KnownFilesStore`** (Менеджер стану): Читає та оновлює локальний словник `known_files.json`. Він гарантує ідемпотентність системи (одне відео - одне завантаження), а також вміє граціозно обробляти биті або порожні JSON-стани.
3. **`Worker`** (Фоновий процес): Нескінченний цикл, що живе в проєкті `TubePilot.Worker`. Він прокидається кожні 30 секунд і, завдяки патерну `IOptionsMonitor`, вміє "на льоту" підхоплювати зміни папок із конфігів без перезапуску програми.

## ⚙️ Конфігурація (`secrets.json`)
Модуль покладається на файл `secrets.json`, який **обов'язково** повинен лежати поруч із запущеним `Worker.exe` (налаштовано через `<CopyToOutputDirectory>`).
Його структура повинна виглядати так:

```json
{
  "GoogleDrive": {
    "FolderId": "ID_ВАШОЇ_ПАПКИ_В_GOOGLE_DRIVE",
    "DownloadDirectory": "абсолютний_шлях_або_просто_downloads",
    "ServiceAccount": {
      "type": "service_account",
      "project_id": "...",
      "private_key_id": "...",
      "private_key": "...",
      "client_email": "...",
      "client_id": "...",
      "auth_uri": "...",
      "token_uri": "...",
      "auth_provider_x509_cert_url": "...",
      "client_x509_cert_url": "..."
    }
  }
}
```
*💡 Фіча для DevOps: Властивість `FolderId` можна змінювати прямо "на гарячу"! Просто перепишіть ID у файлі, натисніть "Зберегти", і `Worker` підхопить нову папку під час наступного циклу.*

## 🧪 Тест-Сценарії (QA Scenarios)

Ось ключові тест-кейси, які ми імплементували для перевірки стійкості (resilience) та відмовостійкості нашої архітектури:

### 🟢 Сценарій 1: Happy Path (Базове завантаження)
- **Дія**: Запустити воркер з правильними ключами в `secrets.json`. Закинути новий файл `.mp4` у розшарену Google Drive папку.
- **Очікування**: Воркер кожні 30 сек пише лог `Polling Google Drive...` Коли він знаходить відео, пише `Downloading...`, зберігає його в дирректорію (дефолт: `downloads/`) і записує ID файлу у масив в кінці файлу `known_files.json`.

### 🟡 Сценарій 2: Idempotency (Тест пам'яті / дублікатів)
- **Дія**: Залишити завантажене відео в Google Drive папці. Перезапустити програму.
- **Очікування**: Воркер відчитає Google Drive, побачить відео, потім звірить його зі своїм `known_files.json`. Він миттєво зрозуміє, що вже качав його, напише Warning: `File [Name] already exists locally, marking as processed` і **пропустить** повторне завантаження.

### 🟡 Сценарій 3: Відсутність або помилковий `FolderId`
- **Дія**: Знавмисне стерти значення `"FolderId"` в `secrets.json` або зробити його порожнім `""`. Запустити програму.
- **Очікування**: `Worker` **не впаде**. Він виведе одне єдине попередження: `GoogleDrive:FolderId is missing... Provide it to enable polling` і просто "засне" у 30-секундному циклі-очікуванні (таким чином дозволяючи іншим модулям бота жити і працювати паралельно).

### 🔴 Сценарій 4: Жорсткий краш (Missing Service Account)
- **Дія**: Повністю видалити блок `"ServiceAccount"` з `secrets.json` або забути додати сам файл `secrets.json` під час деплою.
- **Очікування**: Програма **впаде (Fail-fast)** відразу при старті (в момент ініціалізації DI) з ексепшеном `InvalidOperationException: ServiceAccount configuration is missing.`. Це зроблено свідомо, бо сервіс об'єктивно не може функціонувати без авторизації.

### 🟢 Сценарій 5: Corrupted State (Пошкоджений кеш JSON)
- **Дія**: Відкрити `known_files.json`, повністю видалити весь його текст (файл 0 байт) або навмисно написати там синтаксично неправильний JSON. Запустити програму.
- **Очікування**: Програма **не впаде**. Парсер викине `JsonException`, який буде миттєво перехоплено блоком `catch`. Система напише `Failed to load known files` в лог, обнулить свою оперативну пам'ять списку завантажень і почне працювати з чистого аркуша. Бот врятовано!
